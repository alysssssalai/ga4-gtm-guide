GA4 和 GTM 分別是什麼？

很多人一開始會搞混這兩個工具。先用一句話區分：

工具角色比喻
GA4收資料的倉庫，負責儲存、分析你的數據倉庫
GTM部署代碼的工具，決定何時送什麼資料給 GA4物流排程系統

你可以完全不用 GTM，直接把 GA4 程式碼寫死在程式裡，但這樣每次改追蹤邏輯都要改程式碼、重新部署。GTM 的價值是讓你不動程式碼就能調整追蹤設定。

三者的資料流：

你的平台
Web / App
→
GTM
代碼管理容器
→
GA4
數據儲存 / 報表

GTM 的三個核心概念

所有 GTM 操作都圍繞這三個東西：

① 觸發器 Trigger
什麼時候動作
→
② 變數 Variable
帶什麼資料進去
→
③ 代碼 Tag
做什麼事
✓ 重點
點了購買按鈕（觸發器）→ 帶著 Measurement ID（變數）→ 執行 GA4 代碼（代碼）→ 傳送 purchase 事件給 GA4

SPA 換頁為什麼需要特別處理？

你們的 Web 是 Vue.js SPA，這是追蹤上最容易踩坑的地方。

傳統網站
  • 每次換頁 → 整頁重新載入
  • 瀏覽器重新執行所有 JS
  • GTM 自動重新載入
  • page_view 自動觸發 ✓
SPA（Vue.js）
  • 只有一個頁面殼，內容換掉
  • 換頁不重新載入頁面
  • GTM 不重新執行
  • page_view 不觸發 ✗

SPA 換頁時，雖然頁面沒有重載，但瀏覽器的網址列會悄悄更新，這個變化叫做 History API，Vue Router 就是靠它切換頁面的。GTM 的「歷史記錄變更」觸發器，就是專門監聽這個變化。

所以 SPA 需要兩個觸發器，缺一不可：

觸發器類型負責時機發生幾次
頁面瀏覽 - 初次載入DOM 就緒使用者第一次進入網站每個 session 一次
SPA 換頁 - History Change歷史記錄變更之後每次在 SPA 內切換頁面每換一頁觸發一次
⚠ 注意
每次討論頁面追蹤時，都要主動確認這兩個觸發器都有設定，少任何一個就會有漏洞。

環境隔離策略

測試數據絕對不能污染正式報表，你們採用兩層隔離：

隔離層面工具做法目的
GA4 層2 個 Property測試數據天然進不了 PROD PropertyVIP 數據完全乾淨
GTM 層6 個容器代碼修改在測試容器不影響正式避免未完成代碼上線
Firebase 層2 個 Firebase 專案App 數據對應正確的 GA4 Property正式 / 測試 App 數據分流

事件參數：什麼時候需要請 RD 幫忙？

GTM 自己能讀到
  • 頁面網址、標題
  • 按鈕文字 / CSS class
  • Cookie / localStorage 的值
GTM 讀不到（需 RD 推送）
  • App 內部的 State（交易數量）
  • API 回傳的資料（成交價格）
  • 後端計算結果（訂單 ID）
  • 程式內部變數（VIP 等級）
Q我以為前端只需要一個 GTM 碼，其餘參數跟事件都可以在 GTM 後台設定？

部分正確。GTM 確實可以讀取頁面上已有的資訊，不需要 RD。但交易所的核心數據（成交金額、幣別、訂單 ID）不在 HTML 頁面上，GTM 讀不到。

所以 RD 需要在對應的程式邏輯執行時，主動把這些資料推送給 GTM：

Web：推送 window.dataLayer.push({ event: 'trade_complete', symbol: 'BTC-CAD', price: 50000 })

App：呼叫 FirebaseAnalytics.instance.logEvent(name: 'trade_complete', parameters: {...})

✓ 重點
RD 只需埋設一次，之後你在 GTM / GA4 後台自由調整，不需要再改程式碼。
Q意思是畫面上讀不到的資訊才需要請 RD 推送，畫面上就有就不用？這樣理解對嗎？

對，理解方向正確！但有一個重要補充：「讀得到」不代表「讀到的值是對的」。

問題說明
數字格式不可靠畫面顯示「0.50 BTC」，GTM 讀到的是文字字串，不是數字，無法計算
畫面數據會變動交易所價格每秒更新，GTM 讀取的瞬間不一定是成交當下的值
維護成本高RD 改了 CSS class，GTM 設定的選取器就失效，追蹤默默中斷

所以交易所產品的核心數據，即使畫面上顯示了，還是建議透過 dataLayer 推送，數據才準確、穩定、不怕前端改版。

這章完成後，你會拿到 6 組 Stream ID。PROD 的 3 組給 GTM 和 RD 使用，DEV 的 3 組用於測試驗證。

本章目標架構

🗂 GA4 帳戶（1 個）— 公司帳戶，所有東西都在這裡面
PROD Property
  • 正式環境，真實用戶數據
  • Web Stream（G-PROD-XXXX）
  • iOS Stream
  • Android Stream
DEV Property
  • 測試環境，QA / 開發驗證
  • Web Stream（G-DEV-XXXX）
  • iOS Stream
  • Android Stream
Q為什麼 iOS Stream 和 Android Stream 要分開建？

因為 GA4 的 App Stream 對應的是一個實際存在的 App，iOS 和 Android 在 GA4 / Firebase 的角度是兩個不同的應用程式，必須分開登記。

平台識別方式設定檔
iOSBundle ID（例如 com.yourapp.ios）GoogleService-Info.plist
AndroidPackage Name（例如 com.yourapp）google-services.json
✓ 重點
分開建並不會讓報表變亂。GA4 收到數據時自動標記來源平台，你在報表裡用「平台」篩選就能合併或分開看。
Q如果用 1 個 Property + 多個 Stream 和 2 個 Property 有什麼差別？
比較項目1 Property + 多 Stream2 Properties（你們的選擇）
測試數據隔離靠篩選器，有設錯的風險天然隔離，完全無法混入
開發人員 IP 問題Internal Traffic Filter 可能因 VPN 失效不影響，測試數據本來就在 DEV
VIP 數據品質可能被測試行為污染完全乾淨，適合高精度分析

GTM 代碼設定（Web）

變數 → 觸發器 → 代碼，順序不能反

這章全程在 GTM 介面操作，不需要動任何程式碼。以下步驟要在「Web - 測試」和「Web - 正式」兩個容器各做一次，差別只有 Measurement ID 不同。

⚠ 注意
建議先在「Web - 測試」容器完成並驗證正確後，再到「Web - 正式」容器重複相同步驟。

STEP1
進入 GTM → 「Web - 測試」容器 → 變數 → 新增

變數名稱GA4 Measurement ID
變數類型常數

左側選單 → 代碼 → 新增

App 埋設（Flutter）

Firebase DEV 專案

Step 4：事件規格定義

// Flutter 範例：使用者完成一筆現貨交易 FirebaseAnalytics.instance.logEvent( name: 'trade_complete', parameters: { 'symbol': 'BTC-CAD', // 來自 App State 'side': 'buy', // 買入 or 賣出 'amount': 0.5, // 來自使用者輸入 'price': 50000, // 來自 API 回傳 'currency': 'CAD', }, );
事件名稱觸發時機必要參數平台
login登入成功後method（google / email）Web + App
trade_click點擊買入 / 賣出按鈕symbol, side, amountWeb + App
trade_complete訂單成交後symbol, amount, price, currencyWeb + App
kyc_start進入 KYC 流程第一步step（1 / 2 / 3）Web + App
Q為什麼事件還需要提供必要參數？GTM 碼貼好後不能在後台設定嗎？

在 Web 上，GTM 確實可以自己讀取很多東西。但 App 上每一個自訂事件，RD 都需要在對應的地方加 logEvent() 呼叫，GTM 才能接收到。

關鍵原因是：成交金額、幣別、訂單 ID 這些資料存在 App State 和 API 回傳裡，不在畫面上，GTM 讀不到，必須由 RD 主動推送。

✓ 重點
你定義「要追蹤什麼、參數叫什麼名字」，RD 負責「把呼叫放到正確的地方」。RD 埋好一次後，你在 GTM 後台操作，不需要再找他。

驗證與 Debug

GTM Preview 模式、GA4 DebugView、App Debug 裝置設定
驗證工具總覽
Web 驗證
  • ① GTM Preview 模式：確認觸發器 / 代碼是否啟動
  • ② GA4 DebugView：確認事件是否送達、參數是否正確
App 驗證
  • ① 開啟裝置 Debug 模式（Android adb / iOS Xcode）
  • ② GA4 DebugView：即時看事件流與參數值

Web 驗證：GTM Preview 模式

STEP1
開啟 Preview 模式
進入 GTM → 選「Web - 測試」容器 → 右上角點「預覽」→ 輸入你的測試網址 → 點「連線」。瀏覽器會開啟兩個視窗：你的測試網站，以及 GTM Tag Assistant 面板。

Tag Assistant 確認清單

確認項目正確的狀態如果不對
左側事件列表每次換頁都出現一筆 History Change換了 3 頁只有 1 筆 → 換頁追蹤有問題
Tags Fired（右側）看到「GA4 - 頁面瀏覽」出現出現在 Tags Not Fired → 觸發器沒有正確綁定
Variables（右側）GA4 Measurement ID = G-DEV-XXXX顯示 G-PROD-XXXX → 環境變數設定錯誤
⚠ 常見問題
第一個頁面出現兩筆 page_view。原因是 DOM Ready 和 History Change 在初次載入都觸發了。解法是把 DOM Ready 觸發器的條件限制為「僅初次載入」。

Web 驗證：GA4 DebugView

STEP2
進入 GA4 → 管理 → DebugView
GTM Preview 開啟時，你的瀏覽器就自動變成 Debug 裝置，DebugView 會即時顯示你的每一個動作。
確認項目正確狀態
有沒有收到 page_view每次換頁都應該出現一筆
事件名稱是否正確和你在 GTM 設定的一致
參數值是否正確點開事件，看每個參數的實際值
有沒有重複發送同一個換頁動作不應該出現兩筆 page_view

App 驗證：開啟裝置 Debug 模式

Android（請 RD 操作）

# 開啟 Debug 模式 adb shell setprop debug.firebase.analytics.app com.yourapp # 驗證完關閉 adb shell setprop debug.firebase.analytics.app .none.

iOS（請 RD 操作）

Xcode → Edit Scheme → Arguments → 在「Arguments Passed On Launch」加入：

-FIRDebugEnabled

關閉：移除 -FIRDebugEnabled，重新 build。
